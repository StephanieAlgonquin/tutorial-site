Image Board Studio

Building an Artist Tool

C# .NET 10 Windows Forms SQLite Beginner Friendly

In this tutorial you will build Image Board Studio from scratch — a desktop app for artists to collect visual references and pick colors from images. No coding experience needed. Every step is explained.

What is C#? C# (pronounced "C sharp") is a programming language made by Microsoft. It is one of the most popular languages for building Windows desktop apps.

What is Windows Forms? Windows Forms (WinForms) is a toolkit that comes with C#. It lets you build windows with buttons, panels, and images by writing code instead of dragging things in a designer.

What is SQLite? SQLite is a tiny database that saves data into a single file on your computer — like a spreadsheet, but for your app. No setup or server needed.

ℹ️
This app runs on Windows 10 or 11 only. It uses Windows-specific drawing APIs that are not available on Mac or Linux.

What You'll Build

The finished app has three screens the user can switch between:

  • Start Page — a landing screen with two clickable mode cards
  • Image Board — a large scrollable canvas where you drag, drop, and resize images. Boards are saved to a database so they persist when you close the app.
  • Color Picker — load any image and click on it to sample colors. Build a reusable palette and copy the hex codes.

Part 1 — Data Models

A model is a C# class that describes the shape of your data. Think of it like a form — it defines what fields exist. Open Models/Models.cs and replace all the content with the code below.

1
Define Board and BoardItem

A Board is one named workspace (like a Photoshop document). A BoardItem is a single image card placed on that board, with its position and size.

namespace Imageboard_illust.Models
{
    public class Board
    {
        public int    Id        { get; set; }
        public string Name      { get; set; } = "";
        public string CreatedAt { get; set; } = "";

        // Returns the folder path where this board's images are stored
        public string ImagesDir(string dataDir)
            => Path.Combine(dataDir, "boards", Id.ToString(), "images");
    }

    public class BoardItem
    {
        public int    Id       { get; set; }
        public int    BoardId  { get; set; }
        public string Name     { get; set; } = "";
        public string FileName { get; set; } = "";
        public int    X, Y, Width, Height, ZOrder;
    }
}
💡
{ get; set; } means the property can be read and written from outside the class. This is standard C# for model classes.

Part 2 — Database

Open Database/AppDatabase.cs. This class is responsible for all reading and writing to the SQLite file. It opens a connection when the app starts and keeps it open until the app closes.

1
Create the database file and tables

The constructor finds the user's AppData folder, creates a .db file there, and runs CREATE TABLE IF NOT EXISTS — which creates the tables only if they don't already exist, so it's safe to call every time the app starts.

using Microsoft.Data.Sqlite;
using Imageboard_illust.Models;

public class AppDatabase : IDisposable
{
    private readonly SqliteConnection _conn;
    public string DataDir { get; }

    public AppDatabase()
    {
        DataDir = Path.Combine(
            Environment.GetFolderPath(
                Environment.SpecialFolder.ApplicationData),
            "ImageBoardStudio");
        Directory.CreateDirectory(DataDir);

        _conn = new SqliteConnection(
            $"Data Source={Path.Combine(DataDir, "app.db")}");
        _conn.Open();
        InitSchema();
    }

    private void InitSchema()
    {
        Exec(@"CREATE TABLE IF NOT EXISTS boards (
            id         INTEGER PRIMARY KEY AUTOINCREMENT,
            name       TEXT NOT NULL,
            created_at TEXT DEFAULT (datetime('now'))
        );");
        Exec(@"CREATE TABLE IF NOT EXISTS board_items (
            id       INTEGER PRIMARY KEY AUTOINCREMENT,
            board_id INTEGER NOT NULL,
            name     TEXT, filename TEXT,
            x INT, y INT, w INT, h INT,
            z_order  INT DEFAULT 0
        );");
    }

    // Helper: run a SQL command with no return value
    private void Exec(string sql)
    {
        using var cmd = _conn.CreateCommand();
        cmd.CommandText = sql;
        cmd.ExecuteNonQuery();
    }

    public void Dispose() => _conn.Dispose();
}
2
Add read / write methods

Add these methods to the same class. They let you insert boards and read them back:

public int InsertBoard(string name)
{
    using var cmd = _conn.CreateCommand();
    cmd.CommandText =
        "INSERT INTO boards (name) VALUES ($n); SELECT last_insert_rowid();";
    cmd.Parameters.AddWithValue("$n", name);
    return (int)(long)cmd.ExecuteScalar()!;
}

public List<Board> GetBoards()
{
    var list = new List<Board>();
    using var cmd = _conn.CreateCommand();
    cmd.CommandText =
        "SELECT id, name, created_at FROM boards ORDER BY id;";
    using var r = cmd.ExecuteReader();
    while (r.Read())
        list.Add(new Board {
            Id        = r.GetInt32(0),
            Name      = r.GetString(1),
            CreatedAt = r.GetString(2)
        });
    return list;
}

2
Draw the dot-grid background

The Paint event fires every time the panel needs to redraw. We draw a gradient and a subtle dot grid over it:

Paint += (_, e) =>
{
    using var br = new LinearGradientBrush(
        ClientRectangle,
        Color.FromArgb(22, 22, 28),
        Color.FromArgb(30, 28, 38),
        LinearGradientMode.ForwardDiagonal);
    e.Graphics.FillRectangle(br, ClientRectangle);

    using var dot = new SolidBrush(
        Color.FromArgb(35, 255, 255, 255));
    const int gap = 32;
    for (int x = gap; x < Width;  x += gap)
    for (int y = gap; y < Height; y += gap)
        e.Graphics.FillEllipse(dot, x-1, y-1, 2, 2);
};
💡
After changing a visual property in code, always call Invalidate() on the control to schedule a repaint. Without it, the change won't appear on screen.

Part 4 — Image Item Control

Open ImageItemControl.cs. This is one draggable image card on the canvas. It is a Panel subclass with its own title bar, a close button, and 8-directional resize handles.

Looking at your Solution Explorer, you can see the key members this class contains:

  • Item : BoardItem — the data for this card (position, size, filename)
  • _pictureBox : PictureBox — displays the image
  • _titleBar : Panel — the draggable bar at the top
  • _closeBtn : Button — the × button to remove the card
  • ResizeDir — an enum with directions: None, N, S, E, W, NE, NW, SE, SW
  • DeleteRequested / BringToFrontRequested — events fired to the parent canvas
1
Define the ResizeDir enum

An enum is a named list of options. This one describes which edge or corner the user is dragging to resize:

public enum ResizeDir
{
    None,           // not resizing
    N, S, E, W,     // top, bottom, right, left edges
    NE, NW, SE, SW  // corners
}
2
Handle drag to move

The title bar handles dragging. On MouseDown we save where the drag started; on MouseMove we update the card's position:

private void TitleBar_Down(object? s, MouseEventArgs e)
{
    _isDragging = true;
    _dragStartScreen   = Cursor.Position;
    _dragStartLocation = Location;
}

private void TitleBar_Move(object? s, MouseEventArgs e)
{
    if (!_isDragging) return;
    var delta = new Point(
        Cursor.Position.X - _dragStartScreen.X,
        Cursor.Position.Y - _dragStartScreen.Y);
    Location = new Point(
        _dragStartLocation.X + delta.X,
        _dragStartLocation.Y + delta.Y);
    BringToFrontRequested?.Invoke(this, EventArgs.Empty);
}

Build the scrollable grid canvas
_canvas = new Panel {

_canvas.Paint += (_, e) => {

    int oy = _canvas.AutoScrollPosition.Y % G;
    for (int x = ox; x < _canvas.Width; x += G)
        e.Graphics.DrawLine(pen, x, 0, x, _canvas.Height);

2
Zoom with Ctrl + Scroll
_canvas.MouseWheel += (_, e) => {
    if (ModifierKeys.HasFlag(Keys.Control))
        SetZoom(_zoom + (e.Delta > 0 ? +0.1f : -0.1f));
};

private void SetZoom(float v)
{
    _zoom = Math.Clamp(v, 0.5f, 2.0f);
    foreach (var card in _cards.Values)
    {
        card.Location = new Point(
            (int)(card.Item.X * _zoom),
            (int)(card.Item.Y * _zoom));
        card.Size = new Size(
            (int)(card.Item.Width  * _zoom),
            (int)(card.Item.Height * _zoom));
    }
}

Part 6 — Color Picker

Open ColorPickerPage.cs. It loads an image into a PictureBox and reads the pixel color wherever the mouse is. Your class also has a NativeMethods helper (visible in Solution Explorer) for calling low-level Windows APIs to read screen pixels.

💡
Right-clicking an image card on the Image Board opens a context menu with useful actions. Here's what each option does:
ℹ️
The Crop dialog opens when you click Crop… from the context menu. Drag the handles to adjust the crop area. The rule-of-thirds grid helps with composition. Click Apply Crop when done.
1
Read pixel color on mouse move
private void PicBox_MouseMove(object? s, MouseEventArgs e)
{
    if (_loadedImage == null) return;
    var bmp = (Bitmap)_loadedImage;

    // Convert PictureBox display coordinates → real bitmap pixel
    int bx = (int)(e.X * (float)bmp.Width  / _picBox.Width);
    int by = (int)(e.Y * (float)bmp.Height / _picBox.Height);
    bx = Math.Clamp(bx, 0, bmp.Width  - 1);
    by = Math.Clamp(by, 0, bmp.Height - 1);

    Color c = bmp.GetPixel(bx, by);
    _hoveredSwatch.BackColor = c;
    _hoveredColorLbl.Text    = $"#{c.R:X2}{c.G:X2}{c.B:X2}";
}
⚠️
When SizeMode is Zoom, WinForms adds letterbox padding around the image. For perfectly accurate picks at the edges, subtract the padding offset from e.X and e.Y.

Part 7 — Main Form

Open MainForm.cs. This is the root window of the app — it owns one AppDatabase and acts as a page router. When the user picks a mode, it disposes the current page and loads the new one.

1
Set up the form

Looking at your Solution Explorer, you can see MainForm has exactly these fields: _db, _startPage, _boardPage, _colorPage — and four methods: ShowStartPage(), ShowImageBoard(), ShowColorPicker(), ClearPages().

public class MainForm : Form
{
    private readonly AppDatabase _db = new();
    private StartPage?       _startPage;
    private ImageBoardPage?  _boardPage;
    private ColorPickerPage? _colorPage;

    public MainForm()
    {
        Text        = "Image Board Studio";
        Size        = new Size(1440, 900);
        MinimumSize = new Size(960,  640);
        BackColor   = Color.FromArgb(22, 22, 26);
        ShowStartPage();
    }

    // Always release the database when the window closes
    protected override void OnFormClosed(FormClosedEventArgs e)
        { _db.Dispose(); base.OnFormClosed(e); }
}
2
Implement page switching
private void ShowStartPage()
{
    ClearPages();
    _startPage = new StartPage();
    _startPage.ImageBoardChosen  += (_, _) => ShowImageBoard();
    _startPage.ColorPickerChosen += (_, _) => ShowColorPicker();
    Controls.Add(_startPage);
    Text = "Image Board Studio";
}

private void ShowImageBoard()
{
    ClearPages();
    _boardPage = new ImageBoardPage(_db);
    _boardPage.HomeRequested += (_, _) => ShowStartPage();
    Controls.Add(_boardPage);
    Text = "Image Board Studio — Image Board";
}

private void ClearPages()
{
    foreach (Control c in Controls) c.Dispose();
    Controls.Clear();
    _startPage = null; _boardPage = null; _colorPage = null;
}

Brainstorm — The Problem

Before writing a single line of code, the idea started with a frustration. Artists constantly juggle scattered tools — browser tabs for references, separate apps for color, and no single place that keeps everything together.

01
Too many open tabs
Reference images spread across dozens of browser tabs, slowing the machine and losing context.
02
Color picking is fragmented
Getting an exact color from a reference requires switching apps or imprecise eyeballing.
03
Mood boards don't persist
Browser-based boards disappear when the tab closes. Local tools are too complex or too basic.
04
No offline-first option
Most reference tools require an internet connection and a subscription. Artists offline are left with nothing.
Problem map

Brainstorm — The Idea

The solution was a focused desktop app with exactly two modes. One for organising images. One for extracting colors. No more, no less.

"What if an artist could open one window, drag in references, arrange them visually, and pick colors — all without touching a browser?"

— The one-sentence pitch
App concept

Brainstorm — Why .NET Desktop Development?

Several platforms were considered. .NET WinForms was chosen because it matched every requirement: native speed, local file storage, Win32 API access, and a self-contained executable.

Option Pro Why rejected
✦ .NET WinForms Native speed · SQLite · Win32 APIs · single .exe Chosen — best fit
Electron (JS) Cross-platform, familiar web tech ~150 MB overhead · no native Win32 · slow canvas
WPF (.NET) Modern UI, XAML styling Steeper learning curve for a first project
React (web) Easy to share and deploy No screen eyedropper · no local file system
Python + tkinter Quick to prototype Poor image performance · complex packaging
⚡ Native Windows speed

WinForms renders through GDI+ — the same graphics layer Windows uses. No browser engine in the way.

💾 SQLite is first-class

Microsoft.Data.Sqlite integrates directly. Boards and palettes persist in a single .db file — no server, no cloud.

🪟 Screen-wide eyedropper

Win32 GetPixel and GetDC can sample any pixel on screen — even outside the app window.

📦 Self-contained .exe

.NET 10 apps publish as a single executable. Users double-click and it runs — no installer needed.

Brainstorm — Development Journey

The app was built in six stages, each one unlocking the next. The key decision was to design the data layer first so all UI work had a stable foundation.

Stage 01
Identify the pain point
Noticed the reference workflow was too fragmented. Asked: what is the minimum app that solves this?
Stage 02
Define two focused modes
Decided on Image Board and Color Picker as separate pages. Kept scope tight to avoid feature bloat.
Stage 03
Design the data layer
Defined Board and BoardItem models first. All UI could be built on top of a stable database layer from the start.
Stage 04
Build drag, resize, zoom
ImageItemControl took the most iteration — 8-directional resize handles and zoom-aware coordinates required careful event handling.
Stage 05
Add the screen eyedropper
Win32 APIs turned the color picker from basic into genuinely useful — sampling any pixel on screen in real time.
Stage 06
Polish & export
Added PNG export, per-card crop, inline rename, duplicate color detection, and right-click copy without #.
